0React の核心

まずこの1本の式を頭に入れる

UI = f(state)

「今の状態(state)」を渡すと「表示すべき画面」が返る関数。状態が変われば React が自動で画面を描き直します（再レンダリング）。document.getElementById(...).innerHTML = ... のように DOM を手で書き換える必要がなくなる、これが React の発想です。

1コンポーネント — UI の部品

画面を返す「ただの関数」

src/components/TabBar.tsx
export default function TabBar() {
  const pathname = usePathname();
  return (
    <nav className="app-tabbar">
      <Link href="/">予約する</Link>
      <Link href="/mypage">マイページ</Link>
    </nav>
  );
}
  • 呼ばれると画面(JSX)を return するだけの関数。
  • 関数名は大文字始まり（TabBar）。小文字だと HTML タグ扱いになる。
  • <TabBar /> と書けば他の画面に貼り付けられる。部品の組み合わせで全体を作る。

2JSX — JS の中に HTML を書く記法

見た目そっくりだが実体は JavaScript の式

<Link href="/" className={pathname === "/" ? "active" : ""}>

{ } の中に JS を埋め込める。上は「今の URL がトップなら active クラスを付ける」＝見た目を状態で条件分岐する例で、UI = f(state) そのもの。

HTML との違い: class → className、onclick → onClick。JS の予約語や慣習に合わせている。

3Props — 親から子へのデータ

関数の引数。読み取り専用。

src/components/LiffGate.tsx
export default function LiffGate({
  children,
  requireRegistered = true
}: {
  children: React.ReactNode;
  requireRegistered?: boolean;
}) { ... }
  • 使う側: <LiffGate requireRegistered={false}>...</LiffGate>
  • children は特別な props。タグで囲んだ中身が入る。末尾の return <>{children}</> で「囲んだ中身をそのまま表示」。
  • props は子から書き換え不可。データは常に親 → 子の一方向。

4State と useState — 変化するデータ

React の心臓部

src/components/LiffGate.tsx
const [state, setState] = useState<State>({ kind: "loading" });
  • useState はフック(Hook)。返り値は「現在値」と「更新関数」の2つ。
  • setState(...) を呼ぶと関数が再実行され画面が描き直される。
同ファイル・末尾（UI = f(state) の実例）
if (state.kind === "loading")      return <p>読み込み中...</p>;
if (state.kind === "unregistered") return <p>登録画面に移動...</p>;
return <>{children}</>;
罠: state.kind = "ready" と直接書き換えても画面は変わらない。必ず setState を通す。

5useEffect — レンダリングの外で起きること

API通信・タイマー・購読などの「副作用」

src/components/LiffGate.tsx
useEffect(() => {
  let active = true;
  (async () => {
    await initLiff();
    const res = await authMe(idToken);
    if (!active) return;      // ← 画面が消えた後は setState しない
    setState(...);
  })();
  return () => { active = false; };   // ← クリーンアップ
}, [router, pathname, requireRegistered]);   // ← 依存配列
  • 依存配列の値が変わったときだけ再実行。[] なら初回のみ。
  • クリーンアップ関数（return する関数）はコンポーネントが消える時 or 次の実行前に呼ばれる。ここでは通信完了時に画面が閉じていたら setState しない安全装置。

6イベントハンドラ — 操作に反応する

onClick には「関数を渡す」

src/app/reserve/[date]/page.tsx
<button onClick={() => router.back()}>←</button>

onClick={() => { setMode("CHARTER"); resetSelection(); }}
罠: onClick={router.back()} は「今すぐ実行」になる。() => router.back() と関数で包むことで「クリック時に実行」になる。
操作→ setMode で state 更新→ 再レンダリング→ className が変わり見た目が変化

7制御コンポーネント — フォームの基本

入力欄の値 = state（single source of truth）

src/app/reserve/[date]/page.tsx
<input
  type="text"
  value={groupName}                               // ① state を表示
  onChange={(e) => setGroupName(e.target.value)}  // ② 入力を state へ
/>

「入力欄が値を覚える」のではなく「state が唯一の正解」。だから groupName.trim().length > 0 のように入力値をそのままロジックに使える。select・textarea も同じ（value + onChange）。

8リスト表示と key

.map() で配列を JSX の配列に変換

src/app/reserve/[date]/page.tsx（時刻表）
{Array.from({ length: SLOTS_PER_DAY }, (_, i) => i).map((i) => (
  <button key={i} onClick={() => onTapSlot(i)}> ... </button>
))}
  • JSX 内では for 文が使えない。.map() がループ手段。
  • key が最重要。一意な値を付けると、React が「追加/削除/移動」を効率的に判定できる。付けないと警告＆表示崩れの原因。
ヒント: 本来は index より「データ固有の ID」が理想。ここはスロット番号=index に意味があるので key={i} で妥当。

9useMemo — 派生値の計算とキャッシュ

「持つべき state」と「計算で導ける値」を区別する

src/app/reserve/[date]/page.tsx
const amount = useMemo(() => {
  if (startSlot === null || endSlot === null) return 0;
  if (mode === "CHARTER") return charterPrice(...);
  return freePrice(...);
}, [mode, startSlot, endSlot, headcount, date]);

料金 amount は state から計算で導かれる派生値。別 state に持つと同期ズレのバグになるので計算で求める。useMemo は依存配列が変わった時だけ再計算する最適化。

このファイルの本当の state: courtId, slots, mode, startSlot, endSlot, purpose, groupName, ... だけ。amount, hasSelection, canProceed, dow はすべて計算で導出。これがきれいな React コードの特徴。

10条件付きレンダリング

3つのイディオムを使い分ける

// (a) 早期 return — 画面まるごと切替
if (isPastOrToday) { return (<>...予約不可画面...</>); }

// (b) 三項演算子 — A か B か
{mode === "CHARTER" ? (<label>用途...</label>) : (<label>人数...</label>)}

// (c) && — 条件を満たす時だけ表示
{hasSelection && <p>選択: ...</p>}
{amount > 0 && <p>金額: ...</p>}
罠: {count && <p>...</p>} は count が 0 のとき画面に「0」が出る。amount > 0 && のように必ず真偽値にする。

★フックの2大ルール

ルール理由
① コンポーネント関数のトップレベルでのみ呼ぶ（if / ループの中で呼ばない）React はフックを「呼ばれた順番」で管理している。順番がズレると壊れる。
② React コンポーネント（or 自作フック）の中でのみ呼ぶ

用語まとめ

概念役割コード内の例
コンポーネントUI の部品（関数）TabBar, LiffGate
JSX見た目を書く記法<nav>...</nav>
Props親→子のデータ（読み取り専用）children, requireRegistered
State変化するデータuseState<State>(...)
useEffect副作用（通信・購読）initLiff(), authMe()
イベント操作への反応onClick, onChange
useMemo派生値の計算・キャッシュamount, slotAvail
Next.js 補足: ファイル冒頭の "use client" は「このコンポーネントはブラウザ側で動く（state / effect を使う）」という Next.js 固有の宣言。純粋な React にはない。
